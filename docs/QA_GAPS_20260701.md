# QA Blind-Spot Review & Harness Hardening (2026-07-01)

Purpose: before trusting a green Mayhem board as a field-deployment gate, audit
what the suite **cannot see**. Method: inventory every fault injector and every
oracle the harness has, map them against the failure classes a residential
DERMS actually meets in the field, and close the gaps that are closable in a
software-only bench. This doc records the inventory, the gaps, what was added,
and what remains out of scope (with reasons), so the next review starts from
here instead of from zero.

---

## 1. What the suite already covered (inventory)

**Injectors.** Device write-path lies (`ack_before_effect`, `reject_write`,
`enable_gate`, `ramp_limit`, `wrong_sign`, `soc_refuse`, `charge_disabled`,
`discharge_disabled`), device transport faults (`nan_sentinel`, `latency`,
`exception_code`, `bad_scale` — solar/battery only), meter freeze (`/control
pause`), 7 OCPP charger faults, 10 CSIP content malformations + pricing/curve
attacks, clock offset (`/admin/clock`), 4 MQTT bus faults (via the on-hub
mqttproxy), environment injection (PV W, site load W, SoC).

**Oracles.** Per-scenario diagnosers (constraint / convergence / SOC / stale /
disconnect / recovery / malform-survivability) + a cross-cutting safety audit
(INV-CONNECT, INV-SOC, INV-EXPIRED, INV-EVMAX) that can escalate any verdict,
with settling-ramp and sample-count grace so it can gate deployments without
false-failing on HIL jitter. Ground truth comes from sim `/state`, never from
the hub's own view — the harness can catch the hub being confidently wrong.

**Run modes.** 42 curated scenarios (38 device/CSIP + 4 MQTT), a pairwise
fault matrix (× clock jitter), and a seeded, replayable chaos mode.

That is a strong device-and-content suite. The blind spots were **the WAN, the
installation itself, time as an event (not a wobble), the shape of the
environment, lifecycle edges, and the hub's own mortality.**

## 2. Gaps found → what was built

Ranked by (field probability × harm). All are implemented in this change set.

### Gap 1 — No northbound outage scenario at all (the #1 field event)
Every CSIP scenario assumed the utility server stays reachable and serves
*something*. A residential DERMS loses its WAN constantly. Three distinct
behaviors were untested: enforcing through an outage, expiring a control
**during** an outage (release must come from local clock discipline, not from a
re-walk), and a **wedged** server (accept-then-stall — the killer for a
single-keep-alive-session fetcher like WolfSSLFetcher; a hang without a client
deadline wedges the whole northbound walk).

**Built:** gridsim `POST /admin/outage` (`down` | `hang`, auto-clear
`duration_s`, admin API unaffected) + scenarios **wan-outage-hold**,
**wan-outage-expiry**, **northbound-hang**.

### Gap 2 — Meter CT clamp installed backwards (the #1 install error)
The suite froze the meter and corrupted a solar scale factor, but never lied
about **direction**. A backwards CT is plausible in every sample — only the
sign is wrong — so per-sample range checks can't catch it, and under a
zero-export cap the hub sees a healthy import while the site back-feeds at full
PV: *confidently wrong*, the worst class this suite can produce. Detection
requires an actuation cross-check (command a known change, verify the meter
moves the expected direction), which the hub does not have yet — this scenario
is **expected to FAIL** until it does, and exists to force that feature.

**Built:** meter fault channel (`invert_sign` + the transport set
`nan_sentinel`/`latency`/`exception_code`, previously unreachable on the meter)
+ scenario **meter-ct-inverted** with a dedicated three-way diagnoser
(PASS = held anyway; DEGRADED = flagged/admitted; FAIL = silent).

### Gap 3 — Time as a step event
`clock-jitter` wobbles ±60 s. A real NTP step after holdover, or a DST bug,
jumps the server hours at once — instantly expiring every active control. The
hub must follow server time (CSIP: server clock is authoritative), release, and
uncurtail — without crash or flap.

**Built:** scenario **clock-jump-forward** (+2 h mid-control, judged on clean
release + solar recovery). A backward jump was considered and deliberately
skipped: rewinding server time makes a control legitimately *pre-start* again,
so there is no defensible pass/fail oracle (see §4).

### Gap 4 — Control churn
Utilities supersede controls rapidly during volatile events. Every scenario
posted one control and held it; none exercised rapid mRID turnover — exactly
what stresses adoption latency, fail-closed holds between replacements, and the
breach-guard session logic (the same code the v3 fixes touched).

**Built:** scenario **control-churn** (delete-and-replace the export cap every
~12 s alternating 0 W / 500 W, judged against the looser bound so every
correctly-enforced phase passes).

### Gap 5 — Static environments hide hunting
Every scenario held PV/load constant, so a controller that chases the last
meter sample instead of holding an absolute ceiling could never be caught
oscillating. Hunting wears contactors and inverters in the field and is
invisible to breach-seconds totals (fast oscillation keeps them small).

**Built:** cross-cutting **INV-HUNT** invariant (post-settling breach
re-entries after clear recoveries, hysteresis-banded, ≥3 ⇒ sustained ⇒ demotes
PASS→DEGRADED, runs on **every** scenario) + scenario **pv-flicker**
(cloud-edge sawtooth: PV 100%↔40% every 3 s under a zero-export cap).

### Gap 6 — Faults at the release edge
All faults fired mid-hold. The v3 campaign showed edges are where the hub is
weakest (curtailment-release Mode B). The specific known gap: a cap released
while the inverter is dark leaves the device latched at a stale ceiling that
nobody clears (optimizer-side `restoreOnGenLimitClear` cannot write a dark
device — southbound re-assert-on-reconnect is Phase 4).

**Built:** scenario **release-while-rebooting** (inverter dark via
`exception_code` across the release edge, returns 23 s later). **Expected to
FAIL (or DEGRADED)** until Phase 4 lands; it pins the gap with a verdict so it
can't be forgotten.

### Gap 7 — The hub's own mortality
Watchdog restarts, OOM, power blips, deploys: the always-on controller *will*
die mid-event. Broker loss was covered (mqtt-broker-restart); the orchestrator
process itself was never killed. Recovery must come from retained bus state and
the CSIP re-walk — and shutdown must not emit an un-commanded restore.

**Built:** scenario **hub-restart-mid-cap** (`ssh` → `sudo systemctl restart
lexa-hub` on the hub Pi; BatchMode with a setup probe, so a bench without SSH
key auth reports INCONCLUSIVE instead of a fake verdict).

## 3. Suite shape after this change

- 42 → **51 curated scenarios** (9 new, category tags: Northbound resilience,
  Sensor integrity, Time integrity, Environment dynamics, Hub resilience).
- Safety audit: 4 → **5 cross-cutting invariants** (INV-HUNT).
- New injector surfaces: gridsim `/admin/outage`, meter `POST /fault`.
- Everything discoverable as before: `scripts/mayhem.py --list`, dashboard QA
  tab, `--only id,id` for the new wave, e.g.
  `python3 scripts/mayhem.py --only wan-outage-hold,meter-ct-inverted`.

**Expected first-run impact (honest accounting):** `meter-ct-inverted` and
`release-while-rebooting` are *designed* to fail against today's hub — they
encode missing capabilities (actuation-direction cross-check; reconnect
re-assert), not regressions. `wan-outage-*`, `northbound-hang`,
`hub-restart-mid-cap` are genuinely unknown — they probe code paths
(fail-closed discovery, fetcher deadlines, startup re-adoption) no scenario has
ever exercised. The headline FAIL count will go **up** before it goes down;
that is the point of a blind-spot review.

## 4. Considered and deferred (with reasons)

| Gap | Why deferred |
|---|---|
| Backward clock jump | Rewinding server time makes an expired control legitimately pre-start again; per CSIP the server clock is authoritative, so "correct" behavior is ambiguous — no defensible oracle. Jitter within an event window is covered by clock-jitter. |
| Meter drift/offset (plausible-but-skewed magnitude) | Undetectable in principle without a second reference; the actionable variant (direction) is covered by invert_sign. Revisit if the hub gains an energy-balance estimator worth testing. |
| OCPP out-of-order TransactionEvents / boot-mid-transaction | evsim needs a lifecycle-reorder injector (non-trivial); EV coverage is already the suite's deepest (7 scenarios). Next candidate when evsim is next touched. |
| Modbus unit-ID confusion / register tearing / CRC noise | simonvetter/modbus handles framing; unit-ID and torn-read faults need changes to the shared server layer with lockstep implications (MTR-4). The observable effect (garbage values) is largely covered by nan/bad_scale/invert. |
| DNS-SD flap / wrong server advertised | Discovery happens once at startup on this bench; needs a second gridsim instance to be meaningful. |
| TLS cert rotation / expiry mid-session | Conformance-suite territory (handshake layer), not HIL fault injection; the wolfSSL session model makes mid-session rotation a restart event, which hub-restart-mid-cap now approximates. |
| Wedge detection (hub /status alive but decisions frozen) | High false-positive risk (decisions legitimately go quiet on a stable bench). Reconsider with a hub-side heartbeat counter in /status. |
| Matrix/chaos cells for the new faults | The curated 9 prove the injectors; folding outage/invert into matrix mode doubles run time for pairings whose value is unproven. Add after the first campaign shows which interact. |

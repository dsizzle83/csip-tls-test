# QA Campaign Report — lexa-hub v3 (2026-07-01)

## Overview

10-cycle Mayhem adversarial HIL campaign against the **v3 lexa-hub build**, run on the
69.0.0.x bench in FAST mode. 42 scenarios × 10 cycles = 420 scenario runs. Three prior
builds are tracked for comparison:

| Build | Description |
|---|---|
| v1 | Baseline: old monolithic `cmd/hub` |
| v2 | New distributed `lexa-hub` (northbound + orchestrator + southbound) — several regressions vs v1 |
| v3 | lexa-hub with three targeted fixes (this campaign) |

Cycle data: `scratchpad/qa_cycles/` (v1), `qa_cycles_v2/` (v2), `qa_cycles_v3/` (v3).

---

## V3 Fixes Under Test

Three commits deployed to the hub Pi before this campaign:

1. **Curtailment-release** — `resolve()` in the northbound scheduler now returns a
   `programFound bool`. When `false` (empty or expired program list), the scheduler goes
   fail-closed: holds `lastGood` until `ValidUntil`, then publishes a zero control.
   Previously, the hub silently kept the last commanded setpoint on the device.

2. **worldMoving gate** — `ReadSystemState()` excludes the meter when
   `worldMoving && frozenW(meter)` to avoid false stale-meter detections. When excluded,
   `state.Grid.NetW = NaN`. A side-effect is that `checkImportConvergence` skips
   incrementing `breachTicks` on NaN ticks (it resets to 0), which can delay CannotComply.

3. **battWrongDirTicks** — After 3 consecutive ticks of `chargeCommanded && PowerW > 100W`
   the optimizer force-disconnects the battery. Targets the `battery-wrong-sign` scenario
   where a wiring fault causes the battery to discharge when the hub commands a charge.

## Detailed Analysis: Key Scenarios

### curtailment-release — 3P/0D/7F (v2: 1P/0D/9F)

**Cycle verdicts:** P F P F F P F F F F

**Improvement:** +2 PASS vs v2, but 70% still FAIL.

The `resolve()` fix works in cycles C1, C3, C6 (PASS), but the majority still FAIL.
Two distinct failure modes:

**Mode A — stuck curtailment, no control (C2, C9, C10):** `hub_adopted=False, reacted=True`.
Hub cleared its internal genLimit state (fix worked) but never pushed a corresponding
WMaxLimPct=100% write to the inverter. The device holds its last stale setpoint indefinitely;
solar reports 4000–4800W against 4800W potential — clearly curtailed by an expired command.

**Mode B — transient breach during reboot/reconnect (C4, C5, C7, C8):** `hub_adopted=True,
reacted=True`. Hub holds the control and re-commands curtailment after the device reconnects,
but the breach window during the reconnect transient still exceeds the evaluator threshold.

**Root cause of Mode A:** When the orchestrator transitions from "active genLimit" to
"no control" (programFound=false → control expired → publishNoControl), `applyGenLimitRule`
stops sending curtailment writes, but no uncurtail write (WMaxLimPct=100%) is sent. The
Modbus southbound layer is write-only — it only pushes commands when the orchestrator
actively commands a value, and has no concept of "reset to default on clear". Fix: emit an
explicit WMaxLimPct=100% (or Conn=1 with no limit) when transitioning from a genLimit
control to no-control.

---

### battery-wrong-sign — 0P/2D/8F (v2: 0P/2D/8F)

**Cycle verdicts:** D F F F F F D F F F

**No improvement** from `battWrongDirTicks`. All FAIL cycles show `cannot=True, tail=True,
converge=True` — the breach converges and CannotComply fires correctly. FAIL is triggered
by the **safety audit**: "discharging 4800W at SoC ≤ 10% (reserve floor)". This 5s window
of unsafe operation at t≈47–55s is what overrides the verdict to FAIL.

**Root cause:** `battWrongDirTicks=3` requires 3 consecutive wrong-direction ticks before
force-disconnecting. With the hub's ~10s orchestration cycle, detection takes 20–30s. The
battery SoC passes through the 10% reserve floor during the first 1–2 ticks before the
disconnect lands. The safety violation is a transient that `battWrongDirTicks` catches too
late to prevent.

V1 avoided this 6/10 times because the old monolithic hub's synchronous in-process Modbus
write could disconnect the battery within a single polling cycle (~2s vs ~10–30s for
the distributed hub via MQTT).

**Fix options:**
1. Reduce `battWrongDirTicks` to 1 (risk: false positives on startup transients).
2. Add a fast path: `chargeCommanded && PowerW > 100W && SOC <= SOCReserve+5%` → immediate
   disconnect in the same tick (no tick count needed — reserve proximity makes it unambiguous).

### battery-soc-refuse — 0P/5D/5F (v2: 0P/2D/8F)

**Cycle verdicts:** F D D F F D D D F F

**Improvement:** 5 DEGRADED vs v2's 2. The worldMoving gate fix reduces spurious meter
exclusions, keeping `state.Grid.NetW` valid more often, which allows `impGuard.breachTicks`
to accumulate without reset.

All cycles: `breach=69.5s, reacted=False` (battery physically refuses to discharge — the
hub correctly cannot react). The DEGRADED/FAIL split is purely whether `checkImportConvergence`
reaches `importBreachTicks=5` within the 70s HoldS. It succeeds 50% of the time, making
this a pure timing race on this bench.

The DEGRADED verdict requires `reported_cannot_comply=True`. The FAIL verdict occurs when
`cannot=False` — the breach counter never reached 5 before the window closed. Contributing
factors: MQTT latency between orchestrator ticks, variable Modbus poll timing, and any tick
where netW is NaN still resets the counter.

---

### ack-before-effect — 0P/7D/3F (v2: 0P/7D/3F)

**Cycle verdicts:** F D F D D F D D D D

**Identical to v2.** The v3 fixes don't touch `checkGenLimitConvergence`. The inverter ACKs
the WMaxLimPct write but delays taking effect for 45s (ack_before_effect fault, delay_s=45).
During the 45s delay, the solar W register still reflects actual output (5000W), so
`checkGenLimitConvergence` correctly sees 5000W > 1000W+100W and increments `overCount`.

DEGRADED cycles (7/10): `genBreachTicks=3` is reached within HoldS=50s, CannotComply fires.
FAIL cycles (3/10, C1/C3/C6): the count doesn't reach 3 before the window closes — a
timing race between hub discovery latency and remaining window time.

Both v2 and v3 sit at 70% DEGRADED. This is the architectural floor for this scenario
with the current breach threshold and hub cycle time.

---

### malform-empty-program — 4P/0D/6F (v2: 2P/1D/7F)

**Cycle verdicts:** F F F F F F P P P P

**Marginal improvement:** 4 PASS vs v2's 2. Striking temporal pattern: all FAILs in C1–C6,
all PASSes in C7–C10.

**FAIL mode (C1–C6):** `hub_adopted=True, reacted=True`, breach at t=42–65s running to end
of window. The fail-closed IS working (hub holds exportCap=0W and curtails solar). The
control's `ValidUntil` expires ~42–65s into the 75s observation window; after expiry the
hub correctly releases the expired control, solar uncurtails, and the remaining window
shows a sustained breach.

**PASS mode (C7–C10):** breach=0–5s (tiny, `tail_clean=True`). The hub holds the exportCap
with minimal overshoot. The difference appears to be bench settling: in later cycles the
hub has more stable state at scenario start and the program's effective coverage vs window
timing is more favorable.

**Root cause:** The `postCapProg(holdS=75)` call in the malform scenario posts a program
with a duration shorter than the 75s observation window (setup overhead is absorbed into
the window start). The hub's correct expiry behavior then causes a breach in the remaining
time. This is a **test design issue**, not a hub bug — the hub is correct to release an
expired control. V1 always got DEGRADED (never PASS) because the old hub posted CannotComply
on export breach even after control expiry, which the evaluator accepted as DEGRADED. V3
correctly doesn't post CannotComply for a constraint it no longer holds.

## What the V3 Fixes Actually Fixed

The three targeted fixes had mixed effectiveness:

| Fix | Target scenario | Outcome |
|---|---|---|
| resolve() → programFound | curtailment-release | Partial: 10%→30% PASS. Mode A (stuck curtailment) still present because uncurtail write is never sent. |
| worldMoving gate | stale-meter, battery-soc-refuse | stale-meter: unchanged (was already 0 FAIL). battery-soc-refuse: improved 20%→50% DEGRADED. |
| battWrongDirTicks | battery-wrong-sign | Ineffective: still 80% FAIL. Detects wrong direction too late to prevent SOC-reserve violation. |

Three scenarios that improved significantly without being explicit fix targets:
- `malformed-csip`: 5P/1D/4F → 10P/0D/0F
- `malform-missing-href`: 5P/3D/2F → 10P/0D/0F
- `solar-bad-scale`: 8P/0D/2F → 10P/0D/0F

These likely improved due to incidental changes in the hub's CSIP XML parsing or error
handling between the v2 and v3 builds.

Two scenarios degraded (more FAILs in v3 than v2):
- `clock-jitter`: 3P/3D/4F → 4P/0D/6F (more FAIL, though PASS count similar — DEG→FAIL shift)
- `battery-charge-disabled`: 0P/8D/2F → 0P/6D/4F (DEG→FAIL shift)

Both are HIL-flaky and the degradation is within sampling noise (2 extra FAILs each over
10 cycles is not statistically significant). No code regression is identified.

## Open Issues and Recommended Fixes

### Issue 1: Uncurtail-on-release (curtailment-release Mode A)

**File:** `lexa-hub/internal/orchestrator/optimizer.go` (applyGenLimitRule) and/or
`lexa-hub/internal/southbound/` Modbus command layer.

**Problem:** When the orchestrator transitions from an active genLimit to no control
(maxLimitW = NaN), it stops sending curtailment Modbus writes. No explicit uncurtail command
(WMaxLimPct=100%) is ever written to the inverter. The device keeps its stale setpoint.

**Fix:** In `applyGenLimitRule`, when `math.IsNaN(maxLimitW)` and the previous tick had an
active limit (detectable via a `genGuard.wasActive bool`), emit a WMaxLimPct=100% command
to each connected inverter. Alternatively, the Modbus southbound layer could track the last
commanded setpoint and, on receiving a zero/nil control, proactively restore the inverter
to its uncurtailed state.

### Issue 2: battWrongDirTicks too slow for SOC-reserve safety

**File:** `lexa-hub/internal/orchestrator/optimizer.go` (`checkBatterySafety`)

**Problem:** The 3-tick count (≥20–30s) allows the battery to drain through the reserve
floor before force-disconnect fires.

**Fix:** Add an immediate path in `checkBatterySafety`:
```go
// Immediate disconnect: wrong direction AND at/near reserve — no tick budget.
if chargeCommanded && b.PowerW > complianceBreachW && b.SOC <= o.SOCReserve+5 {
    plan.BatteryCommands[i].ForceDisconnect = true
    plan.AddDecision("safety/batt-wrong-dir-critical", ...)
}
```
Keep the 3-tick path for the normal wrong-direction case (no SOC alarm) to preserve
false-positive resistance.

### Issue 3: CannotComply timing (battery-soc-refuse, ack-before-effect)

**File:** `lexa-hub/internal/orchestrator/optimizer.go`

**Problem:** `importBreachTicks=5` and `genBreachTicks=3` create a timing race vs the
scenario HoldS windows (70s and 50s respectively). With ~10s hub cycles and 5–10s discovery
latency, the window is tight enough to miss 40–50% of the time.

**Options:**
- Reduce `importBreachTicks` from 5 to 3 (matches `genBreachTicks`). Risk: marginally
  noisier import-cap CannotComply on very short load transients.
- Increase HoldS in the Mayhem scenarios for these two cases to 90s, giving the hub more
  room to accumulate ticks.

### Non-issue: malform-empty-program FAIL

The 60% FAIL rate for this scenario is **not a hub bug**. The hub correctly releases an
expired control when ValidUntil passes. The FAIL arises because the Mayhem scenario's
program duration is shorter than the observation window (setup overhead is consumed before
the window starts). Fix is in the test: `postCapProg` should use `holdS + setupPadS` as
the program duration to ensure the control remains valid through the full window.

The distributed lexa-hub architecture introduces timing variability that the old monolithic
hub didn't have. Most high-FAIL scenarios are timing races between the hub's MQTT-based
orchestration cycle and the Mayhem scenario's hold window — not correctness bugs. The two
actionable code fixes from this session are the uncurtail-on-release gap and the
wrong-direction safety fast path.

10-cycle Mayhem adversarial HIL campaign against the bench (69.0.0.x) in FAST mode,
run against the hub build carrying the six 2026-07-02 fixes (northbound fail-open fix +

## Cluster 1: Stuck curtailment after device returns (Phase-4 reconnect gap, still open)

**Scenarios:** `clock-jump-forward` (5/10), `release-while-rebooting` (4/10),
`curtailment-release` (2/10) — 11 FAILs total, all with the identical headline:

> "solar left stuck below potential after the device returned"

Diagnosis text (verbatim across all three scenarios):

> A control set before the dropout was never re-asserted or cleared on the device after
> it re-registered. Customers see lost generation indefinitely after a transient device
> blip.

Example metrics (clock-jump-forward, C01): solar ended the window at 4250 W against 4800 W
potential (`hub_adopted=True, hub_reacted=True, reported_cannot_comply=False`) — the hub
believes it's in the right state and the breach never converges (`breach_converging=False,
converged_at_s=-1`).

**This is the mirror image of the bug the 2026-07-02 fix targeted.** That fix
(`retryDevice` records `lastCtrl` and re-asserts it on reconnect) handles the case where a
control was **active** when the device dropped — it correctly re-pushes that command on
reconnect. It does not handle the case where the control **cleared or expired while the
device was disconnected**: on reconnect, the southbound layer re-asserts the *stale* last
command instead of checking whether a release is now due. The result is functionally the
opposite of curtailment-release Mode A from the V3 report (stuck curtailed) — here the
device comes back holding an old genLimit/exportCap ceiling with no live control backing it,
and nothing clears it.

**Fix:** on reconnect, `retryDevice` (or its caller) needs to check current control state
before blindly re-asserting `lastCtrl` — if no control is active for that device at
reconnect time, issue an uncurtail (WMaxLimPct=100%) instead of replaying the stale command.
This is `lexa-hub/internal/southbound` (device reconnect path) — same file family as the
2026-07-02 Phase-4 fix, needs a companion check.

## Cluster 2: Silent non-convergence, no CannotComply

**Scenarios:** `control-churn` (3/10), `clock-jitter` (3/10), `battery-wrong-sign` (1/10) —
7 FAILs total, all sharing:

> "The hub adopted the control and commanded a correction, but the device did not converge
> to the limit and the hub did NOT post a CannotComply. The hub is asserting control it does
> not actually have."

- **control-churn** (exportCap ≤500W, overshoot up to 3900W/21s): rapid cap rewrites
  (~every 12s) outrun the hub's convergence-verification loop — it keeps commanding but
  never gets far enough into a stable window to detect and report the shortfall before the
  cap changes again.
- **clock-jitter** (exportCap ≤0W, overshoot up to 4480W/34s): CSIP clock corrections during
  an active cap appear to reset or confuse the convergence tracker (`breach_converging=False`
  despite `hub_reacted=True`), similar mechanism to control-churn but triggered by clock
  jumps instead of control rewrites.
- **battery-wrong-sign** (exportCap breach, 4400W peak, 1/10 this campaign — down from the
  8/10 in V2 and 1/10 already achieved in V3/V4): this is the known wrong-direction bug,
  mostly fixed by the V3 `criticalBatteryInversion` fast path. The one FAIL (C08) shows "the
  hub's plan log was empty during the breach — the optimizer recorded no decision about this
  constraint" — a different flavor from the old timing-race failure, suggesting an edge case
  where the fast path doesn't fire (plan never even attempted), not a reversion of the fix.
  Given 9/10 success this campaign, treat as residual noise, not a regression — but worth a
  quick check of what condition suppresses the fast path's decision logging.

**Fix (control-churn/clock-jitter):** the convergence verifier needs a shorter or
churn-tolerant window — either check convergence against the *most recent* control only
(discard stale accumulation across a control rewrite) or lower the threshold so a partial
observation window still triggers CannotComply instead of silently expiring. File:
`lexa-hub/internal/orchestrator` convergence/CannotComply logic (same subsystem as the V3
Issue 3 import-convergence fix, likely the export-side counterpart never got the same
treatment).

## What's Working

- **0 BLIND across 510 scenario runs** — consistent with V4, the hub is never flying blind.
- **2 fully clean cycles** (C06, C09) — when the timing lines up, the hub handles all 51
  scenarios without a single FAIL.
- The 2026-07-02 northbound fail-open, clock-regression guard, and lexa-api expiry fixes
  show no regressions — none of their target scenarios (northbound-hang, wan-outage-hold/
  expiry, general clock handling outside clock-jump-forward/jitter) appear in this
  campaign's FAIL list.
- `battery-wrong-sign` holds at 9/10 — the V3 critical fast path is durable.

## Recommended Next Steps

1. **Cluster 1 fix** (reconnect-time release check) is the highest-value target — it's 11 of
   21 total FAILs (52%) across three scenarios that all reduce to one root cause.
2. **Cluster 2 fix** (churn/jitter-tolerant convergence check) covers another 6 FAILs (29%).
   Together the two clusters explain 17/21 FAILs (81%) — fixing both should bring the
   average toward ~0.4F/cycle.
3. Re-run a shorter validation campaign (3 cycles) after each fix lands, same pattern as the
   2026-07-02 targeted validation, before committing to a full 10-cycle QA_REPORT_V6.
4. Investigate the battery-wrong-sign C08 "empty plan log" anomaly — low priority (1/10),
   but worth a quick trace to confirm the fast path's guard condition isn't silently
   excluding a real case.

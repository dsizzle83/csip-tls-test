# certs/mbaps/ — bench Secure SunSpec Modbus (mbaps) PKI

Generated by `make gen-mbaps-certs` (`scripts/gen-mbaps-certs.sh` →
`cmd/gen-mbaps-certs`, T06.1). Role-bearing client certs for all five mbaps
roles, a device server cert for `sim/mbapsdev`, and the negative-fixture
matrix `internal/mbtls.RoleFromDER` and the future aggregator emulator
(T06.4) / conformance suite (T06.10) test against.

**Regenerate:** `make gen-mbaps-certs`. Destructive-but-deterministic — the
whole tree (including this file, embedded in cmd/gen-mbaps-certs/readme.md)
is rebuilt from scratch with fresh keys/serials every run ("idempotent" =
safe to re-run, not byte-identical output).

**Keys vs. certs:** every `*-key.pem` is gitignored (the repo-wide
`*-key.pem` glob already covers this tree, no `.gitignore` changes needed).
Every `*-cert.pem`, `manifest.json`, and this README are git-tracked, per
the same convention as `certs/client-cert.pem`/`certs/comm004/`.

## Referee independence (T06 PN-1 / T00 ruling C9)

This PKI is minted by the bench's OWN tooling
(`cmd/gen-mbaps-certs`, stdlib `crypto/x509` only) — it does **not** import
`lexa-platform/securemodbus` or any product-side test-PKI generator. The
role-extension OID is a documented copy of `internal/mbtls.RoleOID`
(role.go's own doc comment sanctions this: "a copy of the OID may be reused
for cert MINTING convenience"); the ASSERTION path
(`internal/mbtls.RoleFromDER`) is the bench's independent referee and is
never generated from this tree. Equivalence between the generator's copy and
the real parser is proven empirically by
`internal/mbtls/mbaps_fixtures_test.go`'s `TestMbapsFixtures_RoleFromDER`,
which round-trips every fixture below through the real `RoleFromDER`.

## Tree

```
ca-cert.pem / ca-key.pem                    bench root CA ("nb-mbaps-clients" trust domain)
intermediate-cert.pem / intermediate-key.pem   intermediate CA, signed by root
dev-ca.pem                                  = ca-cert.pem (copy) — trust anchor sim/mbapsdev
                                               loads by default (-ca)
dev-server-cert.pem / dev-server-key.pem    device server leaf (chain: leaf+intermediate),
                                               NO role extension (TCP-28) — sim/mbapsdev's
                                               default -cert/-key
wrong-ca-cert.pem / wrong-ca-key.pem        a second, unrelated root — the untrusted issuer
                                               for the wrong-ca negative
manifest.json                               machine-readable: fixture name -> paths ->
                                               expected RoleFromDER role/error -> chain_valid
clients/<slug>-cert.pem / -key.pem          5 happy-path role clients (chain: leaf+intermediate)
negative/<name>-cert.pem / -key.pem         7 negative fixtures (see below)
```

## Happy path — one client leaf per role

Role ext OID `1.3.6.1.4.1.50316.802.1`, ASN.1 **UTF8String**, exactly one
value (SunSpecTCP-29/30). All EC P-256 (SunSpecTCP-42), EKU
{serverAuth, clientAuth} (a role cert doubles as the southbound identity the
gateway presents to devices — ARCHITECTURE.md §6).

| Role | slug | file |
|---|---|---|
| `GridServiceSunSpec` | grid-service | `clients/grid-service-cert.pem` |
| `SuperAdministratorSunSpec` | super-admin | `clients/super-admin-cert.pem` |
| `NetworkAdministratorSunSpec` | net-admin | `clients/net-admin-cert.pem` |
| `ReadOnlySunSpec` | read-only | `clients/read-only-cert.pem` |
| `LexaVoltReadOnly` | lexavolt-read-only | `clients/lexavolt-read-only-cert.pem` |

## Negative fixture matrix

All negatives are signed by the real intermediate CA (chain-valid,
`RoleFromDER` structurally parses them) **except** `wrong-ca`; both
`wrong-ca` and `expired` are TLS/chain-level rejections, not role-parse
errors — the embedded role is valid in both. This mirrors design doc 01
§3.1: role/authz errors leave the TLS session up; only a bad chain or an
expired cert is a handshake-level failure.

| Fixture | Malformation | `RoleFromDER` result | Handshake |
|---|---|---|---|
| `no-role` | role extension absent | `ErrNoRole` | succeeds |
| `two-role` | two role extensions present | `ErrMultipleRoles` | succeeds |
| `bad-encoding` | role value is PrintableString, not UTF8String | `ErrBadEncoding` | succeeds |
| `empty-role` | role value is the empty UTF8String `""` | `("", nil)` | succeeds |
| `oversize-role` | role value is a 1024-byte UTF8String | `(<1024 A's>, nil)` | succeeds |
| `expired` | `notAfter` in the past, valid `GridServiceSunSpec` role | `("GridServiceSunSpec", nil)` | **rejected** |
| `wrong-ca` | issued by the untrusted `wrong-ca-cert.pem` root, valid role | `("GridServiceSunSpec", nil)` | **rejected** |

**`no-client-cert`** is an eighth documented negative from the T06.1 fixture
matrix with **no cert file**: it drives a handshake where the client
presents no certificate at all (`mbtls.DefaultClientProfile(ca, "", "")`).
Already exercised by `internal/mbtls`'s
`TestHandshake_NoClientCert_Rejected` against the package's own runtime-minted
fixtures; not duplicated here since there is no fixture to mint.

## Verification

- `internal/mbtls/mbaps_fixtures_test.go` (`TestMbapsFixtures_RoleFromDER`,
  no build tag, fast lane): loads `manifest.json` and asserts the real
  `RoleFromDER` against every fixture above.
- `internal/mbtls/mbaps_fixtures_integration_test.go` (`-tags=integration`,
  `make test-integration`): stands up a real `mbtls.Listener` from
  `dev-ca.pem`/`dev-server-cert.pem`/`dev-server-key.pem` (proving the
  device cert loads and serves), dials every happy-path role cert through a
  real wolfSSL handshake and checks the extracted role, and drives every
  negative fixture through the same listener — `wrong-ca`/`expired` must
  fail the handshake, the five role-parse negatives must complete it.
- Cross-checked independently with `openssl verify` / `openssl x509 -text`
  at generation time (the generator's own self-check uses
  `crypto/x509.Certificate.Verify`, a second, stdlib-only verifier).
